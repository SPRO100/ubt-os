## 1. Починено в этой ветке (commit `7ac1b8b`)

### 1.1 CI на main красный с 16:19 сегодня — mypy 226 ошибок ✅
Коммит `1908954` («Bump deploy/requirements.txt…») поднял `supabase` до
2.31.0, где postgrest типизирует `APIResponse.data` как `JSON`
(union из 7 типов) вместо `list[dict]`. Каждое `resp.data[0]["field"]`
стало ошибкой типов → 226 ошибок в 14 файлах, CI падал на каждом пуше
(включая ежечасные «auto sync»). Локально это не воспроизводилось, если
mypy стоял в изолированном venv без supabase — поэтому и просочилось.

**Фикс:** новый модуль `ubt_os/utils/supabase_utils.py` с тремя типобезопасными
хелперами — `rows()`, `first_row()`, `one_row()` — и заменой всех точек
извлечения `.data` в 11 файлах. Поведение в рантайме не меняется (это `cast`).

### 1.2 Реальный баг: проверка прокси всегда «мертва» (A14) ✅
`account_checker.py` создавал `httpx.AsyncClient(proxies={"all://": url})`,
но в httpx 0.28 (который запинен в requirements) аргумент `proxies` **удалён**.
TypeError глотался `except Exception: return None` → каждый прокси молча
считался нерабочим, `last_ping_ms` никогда не обновлялся.
**Фикс:** `proxy=proxy_url`.

### 1.3 Реальный баг: Higgsfield-воркер падал на каждой задаче ✅
`higgsfield_queue.dequeue()`: `zpopmin` возвращает пары `(member, score)`,
а код передавал весь кортеж в `VideoJob.from_json()` → `json.loads(tuple)`
падает. Воркер не мог взять ни одну задачу из очереди.
**Фикс:** берём `items[0][0]` + decode bytes.

### 1.4 main.py: устойчивость webhook-сервера ✅
- Битый JSON в теле → раньше необработанный traceback и обрыв соединения,
  теперь `400 {"error": "invalid JSON body"}`.
- Исключение внутри обработчика роута → раньше n8n получал обрыв соединения
  без ответа, теперь `500 {"error": …}` + `logger.exception`.

### 1.5 Docker-деплой работал без аутентификации ✅
`docker-compose.yml` **не пробрасывал** `WEBHOOK_SECRET` и `AGENTS_API_TOKEN`
в контейнер `agents` → сервер стартовал в dev-режиме и принимал любые запросы
(nginx наружу на :80, только rate-limit). Также не пробрасывались
`PUBLER_API_KEY` (+profile IDs), `FIRECRAWL_API_KEY`, `DEEPGRAM_API_KEY`,
`OPENAI_API_KEY`, `MEDIA_BUCKET`, `TTS_SERVER_URL`, `CORS_ALLOW_ORIGIN` —
то есть A26-публикация, A20-тренды и транскрипция в Docker молча работали
в режиме заглушек. Всё добавлено.

### 1.6 vertical_configs не попадали в Docker-образ ✅
`VerticalLoader` ищет файловый fallback в `/app/vertical_configs`, но
Dockerfile копировал только `ubt_os/`. Добавлен `COPY vertical_configs/`.

## 2. Что ещё починить (по приоритету)

### 🔴 P0 — блокирует прод

1. **Deploy Dashboard workflow красный: не заданы GitHub-секреты.**
   `deploy-dashboard.yml` падает с `error: missing server host` — в репо
   не настроены secrets `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_SSH_KEY`.
   → Settings → Secrets and variables → Actions → добавить все три,
   либо отключить workflow, если деплой дашборда идёт иначе.

2. **`/run/nutra` и `/run/ubt` — пустые заглушки.** Главные видео-пайплайны
   (ради которых существует система) только пишут лог «запуск ✅» и выходят
   (`main.py:187`, TODO). Цепочка A15→A21→A25→A30→A34/A35→A26/social_publisher
   не собрана в пайплайн. Это самый большой недостроенный кусок проекта.

3. **Ежечасные «auto sync» коммиты в main жгут CI.** Каждый час пустой
   коммит триггерит полный CI-прогон (и весь день падал из-за п.1.1).
   → добавить `[skip ci]` в сообщение авто-коммита obsidian-синка, или
   пушить vault в отдельную ветку/репо, или в `ci.yml` добавить
   `paths-ignore: [obsidian-vault/**]`.

### 🟠 P1 — безопасность

4. **`/metrics` и `/health/check-all` публичны без аутентификации** (GET
   в main.py не проверяет auth, nginx проксирует наружу). `/health` при ошибке
   возвращает текст исключения — может утечь строка подключения.
   → либо закрыть эти location в nginx (allow internal), либо проверять Bearer.

5. **Токены платформ в `direct_publish_accounts` — плейнтекст в БД.**
   access_token/refresh_token всех аккаунтов лежат открыто. Минимум — включить
   RLS на таблицу; лучше — шифрование на уровне приложения (Fernet, ключ в env).

6. **Нет ограничения размера тела запроса** в main.py: `Content-Length`
   читается без лимита → DoS большим телом. Добавить лимит (например, 10 MB).

7. **Nginx без TLS** — только :80. Как только будет домен: certbot + 443 +
   HSTS, и `CORS_ALLOW_ORIGIN` сузить с `*` до домена дашборда.

### 🟡 P2 — гигиена

8. **Мёртвые зависимости: `telethon` и `pyrogram`** — не импортируются нигде
   в проекте (A13 telegram_jitter использует только stdlib). При этом pyrogram
   тянет `pyaes`, который **не собирается** на свежих окружениях (сломан
   setup.py) — установка `requirements.txt` с нуля падает. → удалить обе.

9. **`redis[asyncio]==8.0.0`** — extra `asyncio` не существует в redis 8.x
   (pip warning). → заменить на `redis==8.0.0`.

10. **Дрейф деплоя дашборда: два конкурирующих механизма.**
    docker-compose раздаёт `dashboard-static/` (один index.html) через nginx,
    а `deploy-dashboard.yml` + `ubt-dashboard.service` собирают React-апп из
    `dashboard/` через systemd на хосте. Выбрать один путь (логичнее: CI
    собирает `dashboard/dist` → кладёт в volume nginx; systemd-юнит удалить).

11. **CI: Node 20 deprecated** в GitHub Actions (warnings в каждом прогоне),
    а deploy-workflow уже на Node 22. → в `ci.yml` поднять `node-version: 22`.

12. **pytest-asyncio warning**: не задан `asyncio_default_fixture_loop_scope`
    → добавить в `pytest.ini`.

13. **Документация отстала от кода**: CLAUDE.md заявляет «19 агентов
    A12–A30», по факту — 27 модулей до A36. Обновить счётчики и таблицы.
